# Patient Medical Photos

**TL;DR** ‚Äî Authorised staff can attach medical images to a patient record from the prescription page. Images are uploaded via multipart/form-data through an authed fetch, streamed through multer's `memoryStorage` (no disk write), pushed to Wasabi via the [[StorageDriver]] abstraction, and recorded as a `medicalPhotos` subdocument on the patient. The non-obvious part: the patient's existence is checked **before** the upload so a failed DB write can never leave an orphan object in storage.

## The path, in order

1. **UI** ‚Äî [[MedicalPhotos.tsx]] renders the photo grid on the prescription page. Upload button wraps a hidden `<input type="file" multiple accept="image/*">`. On change:
2. **Upload call** ‚Äî `api.upload<{photo}>('/patients/:id/photos', formData)` (added to [[api-client.ts]]). The wrapper sets `X-Skip-Content-Type: 1` so the browser can write its own multipart boundary, then attaches the JWT bearer and runs the standard 401-refresh-retry path.
3. **Route** ‚Äî `POST /api/patients/:id/photos` on [[patients.routes.ts]], guarded by `authenticateToken` + `requirePermission('patients', 'canEditPatients')`, then `patientPhotoUpload.single('file')` from [[upload.middleware.ts]].
4. **Multer** ‚Äî memory storage, 10 MB cap, MIME allowlist (`jpeg`/`png`/`webp`/`gif`). The file buffer lands in `req.file.buffer`; nothing is written to disk.
5. **Controller** ‚Äî [[patients.controller.ts]] `addPatientPhoto` unwraps the buffer and calls [[PatientService.ts]] `addPatientPhoto({ buffer, originalName, contentType, size })`.
6. **Service ‚Äî fail-fast existence check** ‚Äî `PatientService.addPatientPhoto` selects `_id` only to verify the patient exists **before** touching storage. This is the whole reason we can never end up with an orphan object in Wasabi if the `$push` races a delete.
7. **Storage upload** ‚Äî `getStorageDriver().upload(key, buffer, contentType)` where `key = patient-photos/<patientId>/<uuid>.<ext>`. Production uses the Wasabi driver; tests inject the in-memory driver via `setStorageDriver()`. Returns the public URL.
8. **DB write** ‚Äî `$push` into `patient.medicalPhotos`. If the update ever fails (patient was deleted between steps 6 and 8), the service deletes the object we just uploaded in the `catch` block ‚Äî best-effort, no throw-cascade.
9. **Response** ‚Äî `{ photo: { _id, storageKey, originalName, url, contentType, size, uploadedAt } }`. The frontend maps this to its internal `{ id, name, url, uploadedAt }` shape in `toMedicalPhoto()`.

## Schema

Embedded subdocument on [[patient.ts]] ‚Äî kept on the patient so you never need a join to render the prescription page:

```ts
{
  _id: ObjectId,          // generated by mongoose
  storageKey: String,     // 'patient-photos/<pid>/<uuid>.png'
  originalName: String,   // user-provided filename ‚Äî shown in the UI
  url: String,            // absolute https URL returned by the driver
  contentType: String,
  size: Number,           // bytes ‚Äî useful for future storage-quota checks
  uploadedAt: Date,
}
```

The `medicalPhotos` field is explicitly **excluded from `LIST_PROJECTION`** in [[PatientService.ts]] so patient-list endpoints stay lean. A patient with 40 photos at 5 MB each would otherwise bloat every list response with 200 KB of subdoc metadata.

## Delete flow

`DELETE /api/patients/:id/photos?photoId=<id>`:

1. Load patient, find the subdoc by `_id`.
2. `$pull` the subdoc from `medicalPhotos`.
3. `getStorageDriver().delete(storageKey)` ‚Äî fire-and-forget, `.catch(() => {})`. If Wasabi is temporarily unreachable we don't roll back the DB change; the object becomes an orphan that a future sweep can clean.

Both 404 cases (missing patient, missing photo on that patient) come back as proper 404s because of the errorHandler fix described in [[controller-conventions.md]] ¬ß2.

## The storage abstraction

See [[storage-abstraction|Storage Abstraction]] for the driver pattern. Short version: `lib/storage/index.ts` exports `getStorageDriver()` / `setStorageDriver()`; production boots with `wasabiDriver`, tests call `setStorageDriver(createMemoryDriver())` once in the suite.

## Gotchas / open questions

- **`BlobStorageService` is not related.** [[BlobStorageService.ts]] is a vestigial local-file wrapper used by invoice generation. Don't extend it for new features ‚Äî use [[StorageDriver]] instead.
- **Next.js image hosts** ‚Äî Wasabi returns absolute `https://*.wasabisys.com` URLs. The existing `next.config.mjs` allows `protocol: 'https', hostname: '**'`, so images render without further config. If that wildcard is ever tightened, add a scoped `hostname: '*.wasabisys.com'` pattern.
- **No client-side offline cache.** [[usePrescriptionForm]] still reads the initial photo list from `localStorage` keyed by `prescription-photos-<patientId>`. For a cross-device view, the prescription page should also fetch `GET /api/patients/:id/photos` on mount and reconcile. Not wired yet.
- **ACL is `public-read`.** The Wasabi driver sets `ACL: 'public-read'` on upload so `<img src>` works without signed URLs. If you need HIPAA-style protection, swap to presigned GET URLs and update `MedicalPhotos.tsx` to fetch the URL on demand.
- **No server-side rate limit on the upload route.** The global 15-min limiter applies but `bulkOperationRateLimit` does not. A malicious logged-in user can fill the bucket. If this becomes a concern, wrap the route in a tighter per-user limiter.

## Tests

[[patients.test.ts]] ‚Äî the `Medical photo endpoints (Wasabi via memory driver)` suite covers:
- Happy path: upload ‚Üí stores in memory driver ‚Üí subdoc appears in DB with storage-driver-returned URL.
- 400s: no file, non-image MIME, malformed patient id, malformed photoId, missing photoId.
- 404s: patient doesn't exist (and zero objects land in storage ‚Äî orphan check), photoId doesn't exist on patient.
- Delete round-trip: object disappears from both DB and storage-driver map.
- List projection: `GET /api/patients` omits `medicalPhotos` on every row.

Live ops smoke test: `cd backend && npx tsx scripts/smoke-wasabi.ts` ‚Äî requires a working bucket.
